# 긴 노트 — 슬라이딩 윈도우 테스트

## 섹션 1: 배경

이 노트는 슬라이딩 윈도우 청킹이 올바르게 동작하는지 검증하기 위한 긴 텍스트입니다.
chunk_size를 초과하는 섹션이 있어야 슬라이딩 윈도우가 활성화됩니다.
Lorem ipsum 대신 실제 의미 있는 한국어 텍스트를 사용합니다.

RAG 시스템에서 청킹 전략은 검색 품질에 직접적인 영향을 미칩니다.
너무 작은 청크는 컨텍스트가 부족하고, 너무 큰 청크는 노이즈가 많습니다.
적절한 overlap을 설정하면 청크 경계에서 정보 손실을 방지할 수 있습니다.

## 섹션 2: 구현 세부사항

vault.mjs는 Obsidian 마크다운 파일을 읽어 YAML frontmatter를 제거하고,
heading(#, ##, ###) 기준으로 섹션을 분할합니다.
각 섹션이 chunk_size를 초과하면 슬라이딩 윈도우를 적용합니다.

content_hash 필드는 sha256의 첫 8자를 사용하여 delta 인덱싱을 지원합니다.
같은 텍스트는 같은 hash를 가지므로 변경되지 않은 청크는 재임베딩을 건너뛸 수 있습니다.

## 섹션 3: 검색 품질

min_chunk_chars=50 설정으로 너무 짧은 청크를 필터링합니다.
reranker는 짧은 컨텍스트에서 random score를 출력하는 경향이 있으므로
최소 길이 제한이 검색 품질 향상에 도움이 됩니다.

title과 tags를 청크 앞에 prepend하면 retrieval 시 파일 컨텍스트가 함께 전달됩니다.
이를 통해 같은 내용이라도 어떤 파일의 노트인지 모델이 파악할 수 있습니다.
